# 论坛系统数据库字段详细设计文档

> **数据库**：PostgreSQL  
> **设计原则**：大字段与高频字段分离、多态设计、利用唯一索引防并发、预留冗余字段供 Redis/MQ 异步同步。

---

## 一、 用户与账户模块

### 1. `sys_user` (用户主表)
*存储用户基础静态信息，避免与高频变动的资产数据产生行锁竞争。*

| 字段名 | 数据类型 | 约束/默认值 | 中文说明 | 设计巧思与备注 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | BIGSERIAL | PRIMARY KEY | 主键ID | 自增序列。 |
| `username` | VARCHAR(50) | NOT NULL, UNIQUE | 登录账号 | 唯一索引，保证注册不重复。 |
| `password` | VARCHAR(100) | NOT NULL | 登录密码 | 存储加密后的密码（如 BCrypt），不存明文。 |
| `nickname` | VARCHAR(50) | NULL | 用户昵称 | 允许为空，未设置时前端可默认显示 username。 |
| `avatar` | VARCHAR(255) | NULL | 头像URL | 存储图片链接，不存图片本身。 |
| `role` | VARCHAR(20) | DEFAULT 'USER' | 角色权限 | 枚举值：`USER`(普通用户), `ADMIN`(管理员)。 |
| `status` | SMALLINT | DEFAULT 1 | 账号状态 | 1：正常；0：封禁。前端根据此字段控制显示。 |
| `create_time`| TIMESTAMP | DEFAULT CURRENT_TIMESTAMP| 注册时间 | 数据库自动生成。 |
| `update_time`| TIMESTAMP | DEFAULT CURRENT_TIMESTAMP| 更新时间 | 可配合 JPA 的 `@UpdateTimestamp` 自动更新。 |

### 2. `user_stat` (用户统计表)
*与 `sys_user` 形成 1对1 关系，专门存放会被频繁读写的统计数据。*

| 字段名 | 数据类型 | 约束/默认值 | 中文说明 | 设计巧思与备注 |
| :--- | :--- | :--- | :--- | :--- |
| `user_id` | BIGINT | PRIMARY KEY, FK -> sys_user(id) | 用户ID | 既是主键又是外键，强 1对1 关系。 |
| `coin` | BIGINT | DEFAULT 0 | 当前总金币 | **冗余字段**。配合流水表使用，避免每次查余额都去 SUM 流水表。 |
| `post_count`| INT | DEFAULT 0 | 发帖总数 | 冗余字段，发帖时 +1，便于列表快速展示。 |
| `like_count` | INT | DEFAULT 0 | 获赞总数 | 冗余字段，被点赞时 +1。 |
| `sign_count` | INT | DEFAULT 0 | 累计签到天数 | 冗余字段，签到成功后 +1。 |
| `boast_count` | INT | DEFAULT 0 | 参与吹牛次数 | 冗余字段，参与吹牛挑战时 +1。 |
| `boast_win_count` | INT | DEFAULT 0 | 吹牛获胜次数 | 冗余字段，吹牛获胜时 +1。 |
| `boast_win_total` | BIGINT | DEFAULT 0 | 累计赢取金币总额 | 冗余字段，用于"赌神榜"排序。 |
| `boast_best_win` | BIGINT | DEFAULT 0 | 单次最大赢取金币 | 冗余字段，用于"最佳一搏"排行。 |
| `boast_worst_loss` | BIGINT | DEFAULT 0 | 单次最大亏损金币 | 冗余字段，用于"最惨一亏"排行。 |

### 3. `user_wallet_log` (钱包流水表)
*系统绝对安全的“账本”。任何导致金币变动的事件都必须记录于此。*

| 字段名 | 数据类型 | 约束/默认值 | 中文说明 | 设计巧思与备注 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | BIGSERIAL | PRIMARY KEY | 流水ID | |
| `user_id` | BIGINT | NOT NULL, FK -> sys_user(id) | 所属用户 | |
| `amount` | BIGINT | NOT NULL | 变动金额 | 正数为收入（如签到+10），负数为支出（如发帖-5）。 |
| `current_balance`| BIGINT | NOT NULL | 变动后余额 | **核心防错字段**。记录操作后的余额，方便核对是否超发或少发。 |
| `type` | VARCHAR(50) | NOT NULL | 业务类型 | 枚举标识：`SIGN_IN`(签到), `POST`(发帖), `LIKE`(被赞), `RED_PACKET`(抢红包), `BOAST_BET`(下注), `BOAST_WIN`(赢钱)等。 |
| `ref_id` | BIGINT | NULL | 关联业务ID | 记录是哪笔业务产生的流水（如红包表ID）。 |
| `create_time`| TIMESTAMP | DEFAULT CURRENT_TIMESTAMP| 生成时间 | |

## 二、 互动与荣誉模块

### 4. `user_check_in` (签到记录表)
*底层数据防刷保障。*

| 字段名 | 数据类型 | 约束/默认值 | 中文说明 | 设计巧思与备注 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | BIGSERIAL | PRIMARY KEY | 主键ID | |
| `user_id` | BIGINT | NOT NULL, FK -> sys_user(id) | 签到用户 | |
| `check_date`| DATE | NOT NULL | 签到日期 | 只精确到天。 |
| `create_time`| TIMESTAMP | DEFAULT CURRENT_TIMESTAMP| 签到时间 | 精确到秒，可作为同一天重复签到的辅助判断。 |
| **索引** | | **UNIQUE (user_id, check_date)** | | **防并发神键**：数据库底层保证同一用户同一天只能有一条记录，拦截恶意并发点击。 |

### 5. `medal` (勋章表)
*勋章的定义与规则字典表。*

| 字段名 | 数据类型 | 约束/默认值 | 中文说明 | 设计巧思与备注 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | BIGSERIAL | PRIMARY KEY | 勋章ID | |
| `name` | VARCHAR(50) | NOT NULL, UNIQUE | 勋章名称 | 如“签到达人”、“赌神”。 |
| `icon` | VARCHAR(255) | NULL | 勋章图标URL | |
| `description`| VARCHAR(200)| NULL | 获取说明 | |
| `rule_type` | VARCHAR(50) | NULL | 规则类型 | `MANUAL`(需后台手动颁发), `AUTO_STAT`(积分达到自动发)。 |
| `rule_value` | INT | NULL | 规则阈值 | 如 rule_type 为 AUTO_STAT，rule_value 为 100000 表示积分满 10 万自动获取。 |

### 6. `user_medal` (用户勋章关联表)
*多对多中间表。*

| 字段名 | 数据类型 | 约束/默认值 | 中文说明 | 设计巧思与备注 |
| :--- | :--- | :--- | :--- | :--- |
| `user_id` | BIGINT | FK -> sys_user(id) | 用户ID | |
| `medal_id` | BIGINT | FK -> medal(id) | 勋章ID | |
| `create_time`| TIMESTAMP | DEFAULT CURRENT_TIMESTAMP| 获得时间 | |
| **主键** | | **PRIMARY KEY (user_id, medal_id)** | | 联合主键，防止重复颁发同一枚勋章。 |

## 三、 红包模块

### 7. `red_packet` (红包主表)

| 字段名 | 数据类型 | 约束/默认值 | 中文说明 | 设计巧思与备注 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | BIGSERIAL | PRIMARY KEY | 红包ID | |
| `user_id` | BIGINT | NOT NULL, FK -> sys_user(id) | 发红包用户 | |
| `post_id` | BIGINT | NULL, FK -> post(id) | 关联帖子ID | 允许为空，表示可以发“全局红包”，非必须绑定帖子。 |
| `total_amount`| BIGINT | NOT NULL | 红包总金币 | |
| `total_count`| INT | NOT NULL | 红包总个数 | |
| `remain_count`| INT | NOT NULL | 剩余未抢个数 | **高并发热点字段**。抢一个减一个，后期可用 Redis 优化扣减。 |
| `type` | SMALLINT | DEFAULT 1 | 红包类型 | 1：拼手气（随机）；2：普通（平分）。 |
| `status` | SMALLINT | DEFAULT 1 | 状态 | 1：待抢；2：已抢完；3：过期退回。 |
| `create_time`| TIMESTAMP | DEFAULT CURRENT_TIMESTAMP| 发放时间 | |
| `expire_time`| TIMESTAMP | NULL | 过期时间 | 超过此时间未抢完的金币自动退回发起人（可通过定时任务处理）。 |

### 8. `red_packet_record` (抢红包记录表)

| 字段名 | 数据类型 | 约束/默认值 | 中文说明 | 设计巧思与备注 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | BIGSERIAL | PRIMARY KEY | 记录ID | |
| `packet_id` | BIGINT | NOT NULL, FK -> red_packet(id)| 红包ID | |
| `user_id` | BIGINT | NOT NULL, FK -> sys_user(id) | 抢红包用户 | |
| `amount` | BIGINT | NOT NULL | 抢到金额 | |
| `create_time`| TIMESTAMP | DEFAULT CURRENT_TIMESTAMP| 抢夺时间 | |
| **索引** | | **UNIQUE (packet_id, user_id)** | | **防并发神键**：拦截同一用户对同一个红包的多次并发请求（防止“一手快连点抢两次”）。 |

## 四、 吹牛打赌模块 (🔥 特色)

### 9. `boast` (吹牛主表)

| 字段名 | 数据类型 | 约束/默认值 | 中文说明 | 设计巧思与备注 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | BIGSERIAL | PRIMARY KEY | 吹牛ID | |
| `user_id` | BIGINT | NOT NULL, FK -> sys_user(id) | 发布者用户ID | |
| `title` | VARCHAR(150)| NOT NULL | 挑战标题/题目描述 | |
| `image` | VARCHAR(255)| NULL | 截图证据 | 存储图片URL。 |
| `option_one` | VARCHAR(100)| NOT NULL | 选项一描述 | |
| `option_two` | VARCHAR(100)| NOT NULL | 选项二描述 | |
| `correct_option`| SMALLINT | NOT NULL | 正确答案 | 1：选项一为正确答案；2：选项二为正确答案。 |
| `stake_amount`| BIGINT | NOT NULL | 发布者赌注金额 | 发布者发起挑战时投入的金币，挑战者需投入相同金额参与。 |
| `result` | SMALLINT | DEFAULT 0 | 揭晓结果 | 0：进行中；1：发布者获胜（挑战者答错，失去赌注）；2：挑战者获胜（挑战者答对，获得赌注90%）。 |
| `deadline` | TIMESTAMP | NOT NULL | 截止时间 | 前端据此做倒计时展示。 |
| `create_time`| TIMESTAMP | DEFAULT CURRENT_TIMESTAMP| 发起时间 | |

### 10. `boast_bet` (挑战者下注记录表)

| 字段名 | 数据类型 | 约束/默认值 | 中文说明 | 设计巧思与备注 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | BIGSERIAL | PRIMARY KEY | 下注ID | |
| `boast_id` | BIGINT | NOT NULL, FK -> boast(id) | 所属挑战 | |
| `user_id` | BIGINT | NOT NULL, FK -> sys_user(id) | 挑战者用户ID | |
| `option_type`| SMALLINT | NOT NULL | 选择的选项 | 1：选项一；2：选项二。 |
| `amount` | BIGINT | NOT NULL | 下注金额 | 与发布者赌注金额相同。 |
| `result` | SMALLINT | DEFAULT 0 | 结算结果 | 0：未结算；1：赢（答对，获得赌注90%）；2：输（答错，失去赌注）。 |
| `create_time`| TIMESTAMP | DEFAULT CURRENT_TIMESTAMP| 下注时间 | |
| **索引** | | **UNIQUE (boast_id, user_id)** | | 防止同一用户在同一挑战中重复下注。 |

## 五、 内容核心模块

### 11. `category` (板块分类表)

| 字段名 | 数据类型 | 约束/默认值 | 中文说明 | 设计巧思与备注 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | BIGSERIAL | PRIMARY KEY | 板块ID | |
| `name` | VARCHAR(50) | NOT NULL, UNIQUE | 板块名称 | |
| `description`| VARCHAR(200)| NULL | 板块描述 | |
| `sort_order` | INT | DEFAULT 0 | 排序权重 | 数字越小越靠前，前端按此字段 ASC 排序。 |
| `create_time`| TIMESTAMP | DEFAULT CURRENT_TIMESTAMP| 创建时间 | |

### 12. `post` (帖子主表)

| 字段名 | 数据类型 | 约束/默认值 | 中文说明 | 设计巧思与备注 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | BIGSERIAL | PRIMARY KEY | 帖子ID | |
| `user_id` | BIGINT | NOT NULL, FK -> sys_user(id) | 作者ID | |
| `category_id`| BIGINT | NOT NULL, FK -> category(id)| 所属板块 | |
| `title` | VARCHAR(150)| NOT NULL | 帖子标题 | |
| `content` | TEXT | NOT NULL | 帖子正文 | PostgreSQL 的 TEXT 类型无长度限制，适合存长篇图文。 |
| `view_count` | INT | DEFAULT 0 | 浏览量 | **异步冗余字段**。用户浏览时通过 Redis `INCR`，定时同步到此字段。 |
| `like_count` | INT | DEFAULT 0 | 点赞数 | **异步冗余字段**。避免列表查询时联表 COUNT。 |
| `comment_count`| INT | DEFAULT 0 | 评论数 | **MQ异步字段**。有新评论时，发 MQ 异步 +1，保证接口响应极快。 |
| `is_top` | BOOLEAN | DEFAULT FALSE | 是否置顶 | |
| `is_essence` | BOOLEAN | DEFAULT FALSE | 是否加精 | |
| `status` | SMALLINT | DEFAULT 1 | 状态 | 1：正常；0：删除/审核中（软删除，不执行物理 DELETE）。 |
| `create_time`| TIMESTAMP | DEFAULT CURRENT_TIMESTAMP| 发布时间 | 用于按时间倒序排序。 |
| `update_time`| TIMESTAMP | DEFAULT CURRENT_TIMESTAMP| 修改时间 | |

### 13. `tag` (标签表) & 14. `post_tag` (帖子标签关联表)

**`tag` 表：**
| 字段名 | 数据类型 | 约束/默认值 | 中文说明 |
| :--- | :--- | :--- | :--- |
| `id` | BIGSERIAL | PRIMARY KEY | 标签ID |
| `name` | VARCHAR(30) | NOT NULL, UNIQUE | 标签名(如：Java) |

**`post_tag` 表 (多对多中间表)：**
| 字段名 | 数据类型 | 约束/默认值 | 中文说明 | 设计巧思与备注 |
| :--- | :--- | :--- | :--- | :--- |
| `post_id` | BIGINT | FK -> post(id) ON DELETE CASCADE | 帖子ID | 级联删除：帖子删了，关联关系自动清空。 |
| `tag_id` | BIGINT | FK -> tag(id) ON DELETE CASCADE | 标签ID | |
| **主键** | | **PRIMARY KEY (post_id, tag_id)** | | 联合主键防重复打标签。 |

### 15. `comment` (评论表)
*支持无限极楼中楼设计的核心。*

| 字段名 | 数据类型 | 约束/默认值 | 中文说明 | 设计巧思与备注 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | BIGSERIAL | PRIMARY KEY | 评论ID | |
| `post_id` | BIGINT | NOT NULL, FK -> post(id) | 所属帖子 | |
| `user_id` | BIGINT | NOT NULL, FK -> sys_user(id) | 评论者 | |
| `parent_id` | BIGINT | NULL, FK -> comment(id) | 父评论ID | **楼中楼关键**。NULL 表示是对帖子的直接评论；有值表示是对某条评论的回复。 |
| `reply_user_id`| BIGINT | NULL, FK -> sys_user(id) | 被回复者ID | **性能优化**。避免前端为了显示“回复 @张三”而去二次查询父评论是谁发的。 |
| `content` | TEXT | NOT NULL | 评论内容 | |
| `like_count` | INT | DEFAULT 0 | 点赞数 | |
| `status` | SMALLINT | DEFAULT 1 | 状态 | 1：正常；0：删除（软删除，防止楼中楼断裂）。 |
| `create_time`| TIMESTAMP | DEFAULT CURRENT_TIMESTAMP| 评论时间 | |

### 16. `user_like` (通用点赞表)
*使用“多态设计”一张表搞定帖子和评论的点赞。*

| 字段名 | 数据类型 | 约束/默认值 | 中文说明 | 设计巧思与备注 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | BIGSERIAL | PRIMARY KEY | 点赞ID | |
| `user_id` | BIGINT | NOT NULL, FK -> sys_user(id) | 点赞人 | |
| `target_id` | BIGINT | NOT NULL | 目标ID | 帖子ID 或 评论ID。 |
| `target_type`| SMALLINT | NOT NULL | 目标类型 | **多态字段**。1：表示点赞的是帖子；2：表示点赞的是评论。 |
| `create_time`| TIMESTAMP | DEFAULT CURRENT_TIMESTAMP| 点赞时间 | |
| **索引** | | **UNIQUE (user_id, target_id, target_type)**| | **防并发神键**：完美拦截对同一帖子/同一评论的重复点赞请求，无需写分布式锁代码。 |
